# STD-DDL-METADATA-001 — Provenance Metadata Schema

**Status:** Active
**Date:** 2026-04-11
**Author:** Marcus Caldwell (Seat 1002, Claude in app) on behalf of operator
**Authority:** ADR-INGEST-PIPELINE-001
**Applies to:** Every chunk written to any DDL ChromaDB collection

---

## Purpose

Every chunk in every DDL collection carries the same set of metadata fields so that the corpus is queryable by provenance, sortable by ingest time, traceable to its source file, and groupable by content type. Without this contract, the corpus is a soup of unattributed text. With it, the corpus becomes a database with a real schema.

This standard is the contract every ingest path obeys. New ingest paths cannot ship without conforming to this standard. Existing chunks that predate this standard must be backfilled.

---

## Mandatory fields

Every chunk MUST carry all seven of the following metadata fields. No exceptions. No optional fields. No "we'll add it later" — if you can't fill the field at ingest time, use the documented default.

| Field | Type | Required | Default if unknown | Purpose |
|---|---|---|---|---|
| `source_file` | string | yes | — | The filename of the source document, no path |
| `source_path` | string | yes | — | The full absolute path at time of ingest |
| `source_type` | enum | yes | `unknown` | What kind of content this is — see enum below |
| `ingested_at` | ISO 8601 string | yes | — | UTC timestamp when this chunk entered the corpus |
| `ingest_run_id` | string | yes | `manual_YYYY-MM-DD` | Which sweep/manual run created this chunk |
| `chunk_index` | int | yes | — | Position of this chunk within its source file (0-indexed) |
| `chunk_total` | int | yes | — | Total number of chunks from this source file |

### Field semantics

**`source_file`** — Just the filename. `DDLCouncilReview_AntiPractice.txt`, not the full path. Used for grouping and dedup. Case-preserved.

**`source_path`** — The full absolute path to the source file at the moment of ingest. May not match where the file lives later (after sweep moves it to Staging or Archive). This is the *ingest-time* path, not the *current* path. Used for traceability, debugging, and re-ingest matching.

**`source_type`** — One value from the enum below. If the ingest path cannot confidently classify, use `unknown`. Never invent new values without amending this standard.

**`ingested_at`** — ISO 8601 UTC timestamp. Format: `2026-04-11T18:42:18Z`. Always UTC. Always with the `Z` suffix. Generated by the ingest path at the moment of write.

**`ingest_run_id`** — A string identifying the run that created this chunk. Format conventions:
- Sweep runs: `sweep_YYYY-MM-DD_HHMM` (e.g., `sweep_2026-04-11_0300`)
- Manual ingests via dex-ingest-text or similar: `manual_YYYY-MM-DD_HHMM`
- Backfill: `backfill_YYYY-MM-DD`
- Test ingests (against test collections only): `test_YYYY-MM-DD_HHMM_<purpose>`

**`chunk_index`** and **`chunk_total`** — A chunk that is the 12th of 47 chunks from a source file gets `chunk_index: 12` and `chunk_total: 47`. Used for reassembly, completeness checks, and "give me the next chunk in sequence" queries.

## The `source_type` enum

These are the only valid values for `source_type`. Adding a new value requires updating this standard (a new version of STD-DDL-METADATA-001 or an amendment).

| Value | Description | Example sources |
|---|---|---|
| `council_review` | DDL council review documents | `DDLCouncilReview_AntiPractice.txt` |
| `council_synthesis` | Synthesized output from a council review | `SYNTH-CR-DEXJR-RETOOL.md` |
| `governance` | ADRs, standards, protocols, charters | `ADR-INGEST-PIPELINE-001.md`, `STD-DDL-METADATA-001.md`, `PRO-DDL-SPIRAL-001.md` |
| `claude_export` | Saved Claude conversation exports | Thread saves from claude.ai |
| `gpt_export` | Saved ChatGPT conversation exports | Thread saves from chat.openai.com |
| `project_export` | Custom GPT or Claude Project instruction exports | `78_AuditForgePM.txt`, `83_MarcusCaldewellPM.txt` |
| `thread_save` | Generic thread save not from a known LLM source | Manually-saved conversations |
| `transcript` | Text derived from audio via Whisper or similar | Mania voice memo transcripts |
| `prompt` | Saved prompts intended for reuse | `CR-DEXJR-AUDIT-001.txt` (the prompt itself, not the output) |
| `email` | Email content from MBOX, EML, or similar exports | Gmail MBOX-derived chunks |
| `document` | PDF-derived text | Council briefs, dashboards, PDF reports |
| `spreadsheet` | XLSX or CSV-derived text | Audit matrices, FY2025 budgets |
| `web_archive` | Saved web pages — HTML, MHTML, MAFF | Article saves, browser archives |
| `code` | Source code files ingested as text | `.py`, `.jsx`, `.ts`, etc. |
| `system_telemetry` | Sweep reports, audit reports, system self-knowledge | `sweep_2026-04-11_0300.md` |
| `unknown` | Could not be classified — operator review required | Catch-all |

### Inference rules

When the ingest path needs to determine `source_type` automatically, it follows these rules in order. The first match wins.

1. **Filename-prefix rules** (most reliable):
   - `DDLCouncilReview_*` → `council_review`
   - `SYNTH-*`, `*_SYNTH.*` → `council_synthesis`
   - `ADR-*`, `STD-*`, `PRO-*`, `CR-*` (standalone) → `governance`
   - `sweep_*.md`, `audit_*.md`, `*_report.md` → `system_telemetry`

2. **Path-based rules** (when prefix doesn't match):
   - File is in `DDL_Ingest/_reports/` or `DDL_Staging/_reports/` → `system_telemetry`
   - File is in `DDL_Archive/corpus/council_reviews/` → `council_review`
   - File is in `DDL_Archive/corpus/governance/` → `governance`
   - File is in `DDL_Archive/corpus/transcripts/` → `transcript`
   - (etc. — every Archive bucket maps to a default source_type)

3. **Extension-based rules** (when prefix and path are both ambiguous):
   - `.pdf` → `document`
   - `.xlsx`, `.csv` → `spreadsheet`
   - `.html`, `.mhtml`, `.maff` → `web_archive`
   - `.mbox`, `.eml` → `email`
   - `.py`, `.js`, `.jsx`, `.ts`, `.tsx`, `.go`, `.rs` → `code`

4. **Content-based heuristics** (last resort, only if 1-3 don't match):
   - Plain text file with no other signal → `unknown`
   - The ingest path SHOULD log when it falls through to content-based heuristics, so the operator can review and potentially add a more specific rule.

5. **Fallback:** `unknown`. Better to mark a chunk `unknown` and have the operator fix it later than to guess wrong and pollute the type-based queries.

## Validation rules

Every ingest path MUST validate metadata before writing. A chunk that fails validation MUST NOT be written. The ingest path MUST log the failure and the file MUST NOT be moved from `DDL_Ingest` to `DDL_Staging`.

**Validation checks:**

1. All seven mandatory fields are present.
2. `source_type` is a valid enum value (one of the values listed above).
3. `ingested_at` parses as a valid ISO 8601 timestamp with `Z` suffix.
4. `chunk_index >= 0`.
5. `chunk_total >= 1`.
6. `chunk_index < chunk_total`.
7. `source_file` is non-empty.
8. `source_path` is non-empty and is an absolute path (starts with `C:\`, `D:\`, `/`, etc.).
9. `ingest_run_id` is non-empty.

If any check fails, the entire file's ingest is aborted (no partial writes), and the failure is logged with file path and the specific validation that failed.

## Helper function contract

Every ingest path uses a single canonical helper to build the metadata dict. This function lives in `dex_core` (when it exists) or as a standalone module until then. The function signature is:

```python
def build_chunk_metadata(
    source_file: str,
    source_path: str,
    source_type: str,
    ingest_run_id: str,
    chunk_index: int,
    chunk_total: int,
    ingested_at: str | None = None,  # defaults to utcnow() if None
) -> dict:
    """
    Build a validated metadata dict for a single chunk.
    
    Validates all fields per STD-DDL-METADATA-001.
    Raises ValueError on validation failure with a specific message.
    Returns a dict ready to pass to ChromaDB collection.add(metadatas=[...]).
    """
```

No ingest path may construct chunk metadata manually. Every ingest path imports and calls `build_chunk_metadata()`. This is the single point of enforcement for the standard. If the standard changes, the helper changes, and every downstream caller automatically benefits.

## Backfill operation

Existing chunks in the corpus (~542K across 4 collections) predate this standard. They must be backfilled.

The backfill is its own scoped operation, governed by its own pre-flight plan, and is NOT part of the normal ingest pipeline. It runs once.

**Backfill rules:**

1. Set `ingest_run_id` to `backfill_2026-XX-XX` (the date the backfill runs).
2. Set `ingested_at` to the source file's last-modified time if available, otherwise to `2026-04-11T00:00:00Z` (the date this standard was ratified) as a "before this point" marker.
3. Set `source_type` per the inference rules above. If inference falls all the way through to `unknown`, leave it as `unknown` and let the operator review later.
4. Set `chunk_index` and `chunk_total` per actual chunk position in source file, if recoverable. If not recoverable, set both to `0` and flag as `metadata_partial: true` (a backfill-only field).
5. Set `source_file` and `source_path` per existing chunk metadata if any exists, otherwise per inference from chunk content.

**Backfill is allowed to be imperfect.** The goal is "every chunk has the seven fields, even if some values are best-effort." Future audits will surface backfilled chunks that need correction.

## Versioning

This standard is versioned via filename: `STD-DDL-METADATA-001`, `STD-DDL-METADATA-002`, etc. Amendments to a major version are tracked inline with a changelog at the end of the file.

Adding a new `source_type` enum value is a minor amendment — same version, changelog entry.

Removing a `source_type` enum value, changing field names, or changing validation rules is a major amendment — new version supersedes the old, and a backfill operation is required to bring existing chunks into compliance.
